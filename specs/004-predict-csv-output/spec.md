# Feature Specification: Predict CSV Output

**Feature Branch**: `004-predict-csv-output`

**Created**: 2026-05-26

**Status**: Draft

**Input**: User description: "`npm run cli -- predict` should generate `p.csv` file with prediction inside `--input` folder. it should contain: imageId, time, hand, systolic, diastolic, pulse and may be some service info. `npm run cli -- eval` should accept `p.csv` generated by predict command as `--csv` file."

## Clarifications

### Session 2026-05-26

- Q: Which columns should `predict` write to `p.csv` beyond the six core reading fields? → A: Stable service columns: `status`, `confidence`, `uncertainFields`, `provider`, `model`, `rawNotes`
- Q: How should `eval --csv p.csv` interpret a `p.csv` generated by `predict`? → A: Treat `p.csv` as reference rows and compare fresh predictions against it
- Q: When should `predict` write rows to `p.csv` if some images fail? → A: Stream rows as each image is processed
- Q: How should `uncertainFields` be represented in `p.csv`? → A: Encode as a JSON array string in one CSV cell
- Q: How should null or missing prediction values be written in `p.csv`? → A: Write null values as empty CSV cells

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Generate Prediction CSV (Priority: P1)

A CLI user runs prediction for an image directory and receives a `p.csv` file in that same directory as rows are produced, containing one row per processed image with the core reading fields needed for later review or evaluation.

**Why this priority**: This is the primary requested workflow and creates a durable prediction artifact users can inspect, archive, or pass to another command.

**Independent Test**: Can be fully tested by running `predict` against a directory with representative images and confirming `p.csv` is created in the input directory with expected headers and rows.

**Acceptance Scenarios**:

1. **Given** an input directory containing supported image files, **When** the user runs `npm run cli -- predict --input <directory>`, **Then** `<directory>/p.csv` is created with `imageId,time,hand,systolic,diastolic,pulse` columns and receives one prediction row as each image is processed.
2. **Given** a prediction has null or uncertain values, **When** `p.csv` is written, **Then** the row still includes the core columns and stable service columns describing status, confidence, uncertainty, provider, model, and notes.

---

### User Story 2 - Reuse Prediction CSV In Eval (Priority: P2)

A CLI user can pass the `p.csv` generated by `predict` to the existing eval command as the `--csv` file without manual editing; eval treats those rows as the reference values and compares fresh predictions against them.

**Why this priority**: This completes the round-trip workflow requested by the user and prevents duplicated formatting work between commands.

**Independent Test**: Can be tested by generating `p.csv` with `predict`, then running `eval` with `--csv <input-directory>/p.csv` and confirming the command accepts the file and processes the core reading columns.

**Acceptance Scenarios**:

1. **Given** a `p.csv` file generated by `predict`, **When** the user runs `npm run cli -- eval --input <directory> --csv <directory>/p.csv`, **Then** eval accepts the file as reference data, runs fresh predictions for the input images, and compares those fresh predictions against the core reading columns in `p.csv`.
2. **Given** `p.csv` includes extra service information columns, **When** eval reads the file, **Then** those extra columns do not prevent parsing of the core fields.

---

### User Story 3 - Preserve Existing Stream Output (Priority: P3)

A CLI user or automation that currently consumes prediction output can continue to receive prediction records while also getting the new `p.csv` file.

**Why this priority**: Existing workflows should not break when the CSV artifact is added.

**Independent Test**: Can be tested by running `predict` and confirming the usual prediction output is still emitted while `p.csv` is also created.

**Acceptance Scenarios**:

1. **Given** an existing consumer reads prediction output from the command, **When** `predict` completes successfully, **Then** prediction output remains available and `p.csv` is additionally written in the input directory.

---

### Edge Cases

- If the input directory contains no supported images, `p.csv` is still created with headers and no prediction rows.
- If a single image fails during prediction, the CSV receives a row for that image at the point of failure with available core values, nulls for unavailable values, and service information indicating the error.
- If `<input-directory>/p.csv` already exists, the new prediction run replaces it before streaming new rows rather than appending stale rows.
- If the CLI cannot write `p.csv` in the input directory, the command reports a clear failure so the user knows the prediction artifact was not created.
- CSV values containing commas, quotes, or line breaks are escaped so the generated file can be read reliably.
- The `uncertainFields` service column is encoded as a JSON array string in one CSV cell and escaped as needed by CSV rules.
- Null or missing prediction values are written as empty CSV cells.
- Eval rejects CSV files missing any core required column and ignores additional service information columns.

## Architecture & Test Impact *(mandatory)*

- **Ports Affected**: Output writing behavior must support writing a durable prediction CSV artifact in addition to the existing prediction output; evaluation dataset loading must continue to consume core reading columns while tolerating extra columns.
- **Adapters Affected**: CLI/file output adapters and CSV dataset adapter behavior are affected.
- **Boundary Guarantee**: CSV row creation and parsing rules remain independent of framework bootstrap code; command orchestration depends on ports rather than concrete file-system details.
- **Node.js Version Baseline**: Latest active Node.js LTS supported by the project baseline.
- **NestJS Version Baseline**: Latest active NestJS LTS major supported by the project baseline.
- **Dependency Selection Rationale**: Prefer built-in file and stream capabilities for CSV artifact creation unless existing project utilities already provide CSV-safe writing.
- **Existing Test Impact**: Existing prediction and evaluation tests need updates only where they assert command side effects or CSV parsing behavior.
- **New Test Coverage**: Add tests for `predict` creating `p.csv`, replacing existing `p.csv`, including error rows, preserving existing prediction output, and `eval` accepting generated `p.csv` with extra service columns.
- **Coverage Plan**: Maintain CI coverage at or above the project threshold by covering success, empty-directory, write-failure, extra-column, and malformed-CSV cases.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: `predict` MUST create a file named `p.csv` inside the directory provided by `--input` for every prediction run.
- **FR-002**: `p.csv` MUST include the core columns `imageId`, `time`, `hand`, `systolic`, `diastolic`, and `pulse`.
- **FR-003**: `predict` MUST write one `p.csv` row as each input image is processed, including images that produce partial or error predictions.
- **FR-004**: `p.csv` MUST include stable service information columns `status`, `confidence`, `uncertainFields`, `provider`, `model`, and `rawNotes` after the six core reading columns.
- **FR-005**: The stable service information columns MUST NOT be required for eval compatibility beyond the `p.csv` files generated by `predict`.
- **FR-006**: `predict` MUST replace an existing `<input-directory>/p.csv` from a previous run before streaming the current run's predictions.
- **FR-007**: `predict` MUST continue emitting its existing prediction output records while also writing `p.csv`.
- **FR-008**: `eval` MUST accept a `p.csv` generated by `predict` as the value of `--csv` without requiring manual edits, treating it as reference data rather than cached prediction output.
- **FR-009**: `eval` MUST run fresh predictions for the input images and compare them against the core reading columns parsed from the supplied CSV file, even when additional service information columns are present.
- **FR-010**: `eval` MUST reject CSV files that omit any required core column with a clear user-facing error.
- **FR-011**: CSV generation MUST escape values so generated files remain valid when service information includes commas, quotes, or line breaks.
- **FR-012**: If the input directory contains no supported images, `predict` MUST still create `p.csv` with the header row and no data rows.
- **FR-013**: If `p.csv` cannot be written, `predict` MUST report the write failure and avoid presenting the CSV artifact as successfully created.
- **FR-014**: The `uncertainFields` service column MUST encode the uncertain field list as a JSON array string in a single CSV cell, such as `["time","pulse"]`.
- **FR-015**: Null or missing prediction values MUST be written as empty CSV cells, not as literal `null` or placeholder text.
- **FR-016**: Implementation MUST preserve hexagonal boundaries: domain depends on ports only, adapters depend on domain interfaces.
- **FR-017**: Each new feature MUST add new tests; existing tests MUST remain unchanged unless the specification documents why a change is required.
- **FR-018**: Development workflow MUST remain MCP-free and execute in a dedicated feature worktree under `tmp/`.
- **FR-019**: Runtime stack MUST target the latest active Node.js LTS and latest active NestJS LTS.
- **FR-020**: Dependency decisions MUST prefer official Node.js/NestJS modules; third-party additions require explicit justification.

### Key Entities *(include if feature involves data)*

- **Prediction CSV**: The generated `p.csv` file placed in the input directory; contains a header row, six core reading columns, stable service information columns, and receives one row as each image is processed.
- **Prediction Row**: A single prediction result for an image; includes `imageId`, `time`, `hand`, `systolic`, `diastolic`, `pulse`, `status`, `confidence`, `uncertainFields`, `provider`, `model`, and `rawNotes`; any null or missing value is represented as an empty CSV cell.
- **Service Information**: Stable metadata about the prediction process: `status`, `confidence`, `uncertainFields`, `provider`, `model`, and `rawNotes`; `uncertainFields` is a JSON array string in one CSV cell, and service information is useful for troubleshooting but not required for eval compatibility with non-predict CSV files.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% of successful `predict` runs create `<input-directory>/p.csv` with the required core header columns.
- **SC-002**: For a directory containing N supported images, `p.csv` contains exactly N data rows by the end of one completed prediction run, with each row written as its image is processed.
- **SC-003**: 100% of `p.csv` files generated by `predict` are accepted by `eval --csv` without manual editing when the same input directory is used, and eval compares fresh predictions against the CSV rows.
- **SC-004**: Existing prediction output remains available for 100% of successful `predict` runs after the CSV artifact is added.
- **SC-005**: CSV files with extra service information columns are parsed successfully by eval in all covered compatibility tests.
- **SC-006**: Users receive a clear failure message for 100% of tested cases where `p.csv` cannot be written or required CSV columns are missing.

## Assumptions

- The `p.csv` filename is fixed by the requested workflow and is not configurable in this feature.
- `p.csv` is written to the exact directory supplied by `--input`, not to the current working directory or a separate output directory.
- Stable service information columns are always written for observability, but the six core reading columns remain the compatibility contract between `predict` and `eval`.
- Eval treats the CSV passed through `--csv` as the reference comparison dataset and only requires the core reading columns needed for matching and field comparison.
- Existing JSONL-style prediction output remains part of the CLI behavior unless a future feature explicitly changes it.
